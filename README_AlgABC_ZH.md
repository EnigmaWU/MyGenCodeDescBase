# 算法 A、B、C——是什么 & 为什么

## ======>>>共同目标<<<======

三种算法回答的是**同一个问题**：

> 在 `endTime` 时刻的仓库快照里，那些活着的源代码行中，凡是当前版本在 `[startTime, endTime]` 期间引入的，有多大比例归功于 AI？

三种算法的区别只在于**怎么发现行的来源**——度量的东西是一样的。

---

## ======>>>一眼看全貌<<<======

| | **算法 A** | **算法 B** | **算法 C** |
|---|---|---|---|
| **核心手段** | 实时跑 `git`/`svn` blame | 离线 diff 回放 | genCodeDesc 里内嵌的 blame |
| **运行时要不要访问仓库** | 要 | 不要 | 不要 |
| **用的 genCodeDesc 版本** | v26.03 | v26.03 | v26.04 |
| **需不需要逐提交的 diff 补丁** | 不需要 | 需要 | 不需要 |
| **正确性的权威来源** | VCS blame（最高） | 重建的部分 blame（中等） | codeAgent 写入时的数据（信任制） |
| **投产状态** | 生产质量 | 窄路径可用 | 计划中 |

---

## ======>>>算法 A——基于 Blame 的终态快照归属<<<======

### 它是什么

算法 A 是**首选的、生产质量的基线方案**。它从 `endTime` 时刻的活文件快照出发，对每一行跑 `git blame` 或 `svn blame`，用 blame 的结果来发现哪个提交最后引入了这行的当前版本。来源提交落在 `[startTime, endTime]` 窗口里的行就是要度量的。对每一行，去对应的 v26.03 genCodeDesc 记录里查 `genRatio`。

### 为什么它行

- **直接回答** P0 度量问题，基于活快照。
- 改名和移动的检测靠的是**成熟的 VCS blame 实现**，不用自己造。
- 逻辑风险低：blame 是**权威的**行来源——不需要部分重建。
- Git 和 SVN 都能用。

### 已知坑

- 需要**活的仓库访问**——运行时必须有本地检出。
- 在**非常大的仓库**里 blame 性能可能很慢，文件多、文件大的时候尤其明显。
- 正确性取决于 VCS blame 的质量——SVN 碰上复杂的 mergeinfo 可能返回不精确的结果。
- blame 发现的每个来源版本都得有对应的 v26.03 文件。

---

## ======>>>算法 B——不用 Blame 的增量式血统重建<<<======

### 它是什么

算法 B 回放一组按顺序排列的**提交 diff 补丁**（`commitDiffSet`），增量式地重建行的所有权。它不问 VCS "谁最后改了这一行？"，而是模拟历史——按顺序打 diff，追踪哪个提交引入了每一行活着的行。**运行时不需要访问活仓库**。

### 为什么它存在

- 能做到**离线分析**，不需要活的仓库检出。
- blame 跑太慢或者根本用不了的时候很有用。
- diff 产物可以**预索引、便宜地查询**。
- 能算出 blame 做不到的**历史过程度量**（比如写了又删的 AI 行、代码翻动率、存活率）。
- 在测试环境里能做**确定性重放**。

### 已知坑

- 实际上是在**重建一个部分 blame 引擎**——回放逻辑有任何缺口都会悄悄产生错误的归属。
- 每个被回放的版本都得有一个 unified-diff 补丁文件。每个 patch 文件代表完整提交 diff：可以覆盖多个文件，每个文件 diff 又可以包含多个 hunk。
- 合并感知的血统回放**很复杂**——合并多的历史要达到生产就绪需要认真做 TDD。
- SVN 的 path-copy 和 mergeinfo 语义引入的回放边界情况还没完全覆盖。
- 还是需要逐版本的 genCodeDesc v26.03——只是把 blame 那步省了。

---

## ======>>>算法 C——内嵌 Blame，纯 genCodeDesc<<<======

### 它是什么

算法 C 是一个计划中的离线算法，运行时**不需要访问仓库**，也**不需要 diff 产物**。codeAgent 只记录每次提交新增或删除的行，每行带上真实的 VCS blame 信息，写进一个 `genCodeDescProtoV26.04.json` 文件。因为每个 add 条目带着内嵌的 blame（`revisionId`、`originalFilePath`、`originalLine`、`timestamp`），下游消费者只要把所有文件按时间戳顺序累积出完整的存活行集合，然后用 `[startTime, endTime]` 过滤，直接读 `genRatio` 就行了。

### 为什么它存在

- 分析时**零 VCS 访问**——不用检出、不用子进程、不用网络。
- **零 diff 产物**——不需要 commitDiffSet。
- 每次提交的文件很小：只记变更的行，不是全量快照。
- Git 和 SVN 来源的 blame 都能用（VCS 类型是嵌进去的元数据）。
- 最适合**断网、边缘设备、大规模批处理**场景。

### 已知坑

- 必须处理**从头到 endRevision 的每一个提交文件**——链里缺了一个文件，结果就废了。
- `REPOSITORY.revisionTimestamp` 是**必须有的**，处理顺序靠它。
- Delete 条目必须引用**精确的 blame 来源**——不匹配的话会悄悄留下幽灵行。
- 嵌进去的 blame 必须是**真实的 VCS blame**，写入时抓的。合成的、推断的、手动编辑的 blame 会打破契约。
- 分析时没法用 VCS 独立验证——**正确性完全信任 codeAgent**。
- 如果 genCodeDesc 写了之后发生了 force-push 或 amend，嵌进去的 blame 就**悄悄过时了**。

## ======>>>不可替代的优势<<<======

每种算法有一个别人**替代不了**的东西：

- **算法 A** 在**权威性和可追责性**上不可替代——它是唯一直接依赖活 VCS blame 作为权威事实源的，能追溯回原始仓库证据。
- **算法 B** 在**基于补丁的历史重放**上不可替代——从同样的补丁产物做确定性重执行、历史窗口实验、过程重建，这些 A 和 C 做不到。
- **算法 C** 在**最小运行时依赖的离线可伸缩性**上不可替代——它同时做到了零仓库访问和零 diff 回放，这个部署优势其他两个做不到。

## ======>>>附录：blame 是什么<<<======

三种算法都依赖 **blame** 这个概念——对文件里的任意一行，你可以问：**哪个版本最后引入了这一行的当前文本内容？**

blame 是**逐行的**，不是逐提交的。一个有 3 行的文件里，每行可能来自不同的版本：

```
文件在 commit abc123 时：
  第 1 行: "int x = 0;"   → blame: revision 111aaa（3 个月前）
  第 2 行: "x += 1;"      → blame: revision 222bbb（1 周前）
  第 3 行: "return x;"    → blame: revision abc123（这次提交）
```

这就是为什么 blame 天然能处理**改名**（穿透文件路径变化追踪）、**合并**（穿透合并分支追踪）、和**重写**（指向更新的版本）。

每种算法怎么获取 blame：

| 算法 | blame 来源 |
|---|---|
| A | 分析时实时跑 `git blame` / `svn blame` |
| B | 通过回放 diff 重建的（重建的部分 blame） |
| C | codeAgent 写入时嵌进 v26.04 里的 |
